Cross Propagation All-Reduce 구현 정리 (v2)

1. 연구 배경 및 구현 목적

본 구현은 BookSim 2 기반의 cycle-accurate interconnection network simulator를 확장하여,
5x5 mesh 환경에서 Cross Propagation All-Reduce 통신 패턴을 모델링하기 위해 수행되었다.
기존 BookSim은 범용적인 네트워크 토폴로지, 라우팅 알고리즘, 라우터 마이크로아키텍처 실험을
지원하지만, 특정 collective communication 알고리즘의 계층적 데이터 전파 구조를 직접적으로
표현하기에는 한계가 있다. 이에 따라 본 연구에서는 cross 형태의 중심 전파 구조를 활용하는
전용 all-reduce 모델을 BookSim 내부에 통합하였다.

본 구현의 목표는 실제 연산 결과값을 계산하는 기능을 추가하는 것이 아니라, Cross Propagation
All-Reduce의 통신 구조와 단계별 지연 특성을 cycle 수준에서 관찰할 수 있도록 하는 데 있다.
즉, 각 단계에서 필요한 패킷이 모두 도착하면 이를 집계 완료로 간주하고 다음 단계 패킷을
주입하는 방식으로 collective 동작을 모델링하였다.

2. Cross Propagation All-Reduce의 기본 구조

본 구현은 5x5 mesh를 전제로 하며, 중앙 좌표는 k/2 = 2로 고정된다. 이에 따라 전체 노드는
다음과 같이 역할별로 구분된다.

- Center node: N12
- Column-cross node: N2, N7, N17, N22
- Row-cross node: N10, N11, N13, N14
- Outer node: 나머지 16개 노드

이 구조는 대칭적 all-reduce 구현과 다르게 의도적으로 비대칭성을 가진다. 그 이유는
column-cross 노드들이 outer 노드와 동일한 row 축을 공유하므로 Phase 4에서 수평 방향
broadcast를 자연스럽게 담당할 수 있기 때문이다. 마찬가지로 row-cross 노드들은 outer 노드와
동일한 column 축을 공유하므로 수직 방향 broadcast를 담당한다. 이 두 역할의 분리로 인해
Phase 4에서 수평 링크와 수직 링크가 동시에 활용된다.

3. 단계별 통신 동작

Cross Propagation All-Reduce는 TrafficManager 내부의 상태 머신을 기반으로 총 4개의 단계로
구성된다. 각 단계는 Phase 1, 2, 3, 4로 명명된다.

첫 번째 단계(Phase 1)는 Outer to Cross 단계이다. outer 노드들은 cross 구조를 향해 패킷을
전송한다. 각 outer 노드는 두 방향으로 D/2 크기의 패킷을 주입한다. Px 패킷은 XY routing을
사용하여 동일한 row의 column-cross 노드로, Py 패킷은 YX routing을 사용하여 동일한 column의
row-cross 노드로 전송된다. is_px 플래그는 flit 내부에 저장되며 패킷 생애주기 동안 변경되지
않는다.

두 번째 단계(Phase 2)는 Cross to Center 단계이다. 각 cross 노드는 자신에게 할당된 outer
노드 4개로부터 패킷을 모두 수신한 후(로컬 배리어 충족), D/2 크기의 집계 패킷을 center
노드 N12로 전송한다. center는 8개 cross 노드로부터 패킷을 모두 수신하면(글로벌 배리어 충족)
다음 단계로 전이한다. 여기서 중요한 점은 BookSim 내부에서 실제 합산 연산이 수행되지 않는다는
것이다. "reduction"은 정해진 수의 패킷 수신을 집계 완료로 간주하는 제어 흐름으로 표현된다.

세 번째 단계(Phase 3)는 Center to Cross 단계이다. center는 글로벌 배리어 완료 후 모든
8개 cross 노드에 D/2 크기의 패킷을 주입한다. 이 단계는 중심 집계 결과가 cross spine으로
다시 확산되는 과정을 나타낸다. center의 4방향 injection slot(N/S/E/W)을 통해 2 injection
cycle에 걸쳐 8개 패킷이 주입된다.

네 번째 단계(Phase 4)는 Cross to Outer 단계이다. v2에서 설계가 변경된 핵심 단계이다.
column-cross와 row-cross가 모두 참여하여 링크 활용도를 최대화한다.

- Phase 4a: column-cross 노드가 D/2 Px 패킷을 동일한 row의 outer 노드 4개에게 수평 방향으로 broadcast한다.
- Phase 4b: row-cross 노드가 D/2 Py 패킷을 동일한 column의 outer 노드 4개에게 수직 방향으로 broadcast한다.

각 outer 노드는 column-cross로부터 D/2 Px와 row-cross로부터 D/2 Py를 수신하여 합산하면
최종 결과 D를 얻는다. 두 broadcast가 서로 다른 링크(수평/수직)를 사용하므로 충돌 없이
병렬로 진행된다. Phase 4의 종료 배리어는 총 32개 패킷 수신(Px 16개 + Py 16개)으로 설정된다.

4. 네트워크 구조 변경 사항

기존 BookSim의 mesh topology는 사실상 KNCube 계열 구현을 재사용하는 형태였으나, 본 연구에서는
Cross Propagation 실험을 위해 별도의 Mesh 클래스를 새로 정의하였다. 이 클래스는 단순히 이름만
분리한 것이 아니라, 노드 역할 분류와 비균일 injection bandwidth, fat-bus 형태의 병렬 채널 구성을
직접 제어하도록 설계되었다.

구체적으로 outer node는 2개의 injection slot, cross node는 1개의 injection slot, center node는
4개의 injection slot을 갖도록 하였다. 또한 cross 링크는 2개의 병렬 채널, center로 연결되는 링크는
4개의 병렬 채널을 갖도록 구성하여 중심부 대역폭을 확장하였다. 이를 통해 계층적 집계 및 재전파가
수행되는 spine 구간에서 병목을 줄이고자 하였다.

이와 함께 기존 KNCube 구현에는 defer_build 경로를 추가하여, Mesh 클래스가 네트워크 구축 과정을
직접 수행할 수 있도록 수정하였다. 또한 topology가 "mesh"로 지정될 경우 더 이상 기본 KNCube가
아니라 새로 정의한 Mesh 클래스가 생성되도록 network.cpp를 수정하였다.

5. 라우팅 함수 및 패킷 메타데이터 변경

Cross Propagation 경로를 지원하기 위해 routefunc.cpp에 cross_propagation_dor_mesh 라우팅 함수를
추가하였다. 이 함수는 is_px 플래그를 기반으로 XY 또는 YX 경로를 선택하며, fat-bus 링크에서
병렬 포트가 존재하는 경우 방향별 base offset을 사용해 적절한 출력 포트를 계산한다.

이를 위해 flit 구조체에는 is_px 필드가 새롭게 추가되었으며, flit 초기화 시에는 false로 reset되도록
수정하였다. 결과적으로 각 패킷은 주입 시점에 방향 성격이 고정되며, 시뮬레이션 도중 라우팅 방향이
변하지 않는다.

6. TrafficManager 중심의 구현 방식

본 구현에서 가장 큰 변화는 TrafficManager의 역할 확장이다. 기존 BookSim은 traffic pattern에 따라
패킷이 생성되고 네트워크를 통과하는 일반적 구조를 가진다. 반면 이번 구현에서는 CP All-Reduce의
계층적 단계 전이를 정확히 표현하기 위해 TrafficManager가 사실상 전체 통신 절차를 직접 제어한다.

이를 위해 CP 전용 상태 변수, cross 노드 수신 추적 배열, 노드 역할 판별 함수, 단계별 패킷 주입 함수,
배리어 기반 전이 로직 등이 추가되었다. 특히 _RetireFlit()에서 수신 완료 수를 집계해 다음 단계로
넘어가는 구조는 본 구현의 핵심 제어 메커니즘이다.

또한 CP 모드에서는 기존 BookSim의 일반 통계 출력이 알고리즘 해석에 큰 의미를 주지 않거나 오히려
혼선을 줄 수 있으므로, DisplayStats() 및 DisplayOverallStats()의 일부 출력을 억제하는 방향으로
수정하였다.

7. 보조 인프라 변경

Fat-bus 기반의 비균일 채널 구성을 지원하기 위해 buffer, buffer_state, iq_router 계층에도 일부
접근자 함수가 추가되었다. 예를 들어 Buffer의 크기를 동적으로 조회하거나 변경하는 함수, VC buffer
크기를 조정하는 함수, 라우터 내부 input buffer와 output buffer state에 접근하는 함수 등이 추가되었다.
이 변경은 알고리즘 자체보다는, center 및 cross 구간에 필요한 병렬 대역폭 구성을 실현하기 위한
기반 작업에 해당한다.

또한 traffic pattern 계층에는 CrossPropagationTrafficPattern 클래스가 추가되었으나, 현재 구현에서
실질적인 패킷 주입은 TrafficManager가 직접 제어하므로 이 클래스는 보조적이거나 placeholder 성격이
강하다.

8. v1 대비 변경 사항 (v2)

v2에서는 다음 두 가지 변경이 적용되었다.

첫째, 단계 명칭을 정비하였다. 기존에 Phase 0, Phase 0.5, Phase 1, Phase 1.5로 혼재되던 명칭을
Phase 1, Phase 2, Phase 3, Phase 4로 통일하였다. 코드 내 상태 변수, 주석, 로그 출력 모두 새 명칭
기준으로 일관되게 수정하였다.

둘째, Phase 4 설계를 변경하였다. 기존 구현에서는 column-cross 노드만 outer 노드로 broadcast를
수행하고, row-cross 노드는 Phase 3에서 데이터를 수신한 후 종단 노드로 동작하였다. 이 구조에서는
수직 링크(center column의 N/S 방향)가 Phase 4 동안 유휴 상태가 되는 문제가 있었다. 또한
column-cross가 full D 크기의 패킷 4개를 단일 injection slot을 통해 순차적으로 주입하여 직렬화
지연이 발생하였다.

v2에서는 Phase 4를 다음과 같이 재설계하였다.

- Phase 4a: column-cross가 D/2 Px 패킷을 동일 row의 outer 노드 4개에게 수평 방향으로 broadcast
- Phase 4b: row-cross가 D/2 Py 패킷을 동일 column의 outer 노드 4개에게 수직 방향으로 broadcast

이 설계는 Reduce-Scatter 단계(Phase 1)와 대칭적인 구조를 가진다. Phase 1에서 outer가 col-cross로
Px를, row-cross로 Py를 보냈던 것의 역방향이다. 결과적으로 수평 링크와 수직 링크가 Phase 4에서
동시에 활용된다.

아울러 Phase 3에서 center가 각 cross 노드로 전송하는 패킷 크기를 D에서 D/2로 변경하였다.
각 cross 노드는 자신이 받은 D/2를 그대로 outer 4개에게 broadcast하는 relay 구조를 유지한다.

이 변경으로 인한 시뮬레이션 결과 비교는 다음과 같다.

  ┌──────────────────────┬──────────────┬──────────────┐
  │                      │   v1 (구 설계) │   v2 (새 설계) │
  ├──────────────────────┼──────────────┼──────────────┤
  │ Reduce  (Phase 1+2)  │   67 cycles  │   67 cycles  │
  ├──────────────────────┼──────────────┼──────────────┤
  │ Broadcast (Phase 3+4)│   76 cycles  │   56 cycles  │
  ├──────────────────────┼──────────────┼──────────────┤
  │ Total All-Reduce     │  143 cycles  │  123 cycles  │
  ├──────────────────────┼──────────────┼──────────────┤
  │ Total flits injected │  256 flits   │  320 flits   │
  └──────────────────────┴──────────────┴──────────────┘

Broadcast latency가 76에서 56 cycles로 20 cycles 단축되었다. 총 flits 수가 증가한 이유는
row-cross의 Phase 4b broadcast가 추가되었기 때문이다(4 row-cross × 4 outer × D/2 = 64 flits 추가).

9. 현재 구현의 한계

첫째, 현재 구현은 5x5 mesh에 강하게 특화되어 있다. center index가 2로 고정되어 있으며,
배리어 threshold 값 4, 8, 32 및 cross 추적 배열 크기 8 역시 모두 5x5 구조를 전제로 한다.
따라서 k 값을 일반화하려면 노드 역할 분류, 예상 패킷 수 계산, 단계 전이 조건을 모두 다시
파라미터화해야 한다.

둘째, PE computation delay가 모델링되어 있지 않다. 본 시뮬레이션은 통신 시간 중심으로
동작하며, 각 processing element 내부에서 발생하는 local reduction 연산 시간은 반영하지 않는다.
비교 대상인 TTO도 동일하게 relay PE overhead를 모델링하지 않으므로, BookSim 기반의 상호 비교는
공정하게 수행될 수 있다.

셋째, Phase 4에서 각 cross 노드는 단일 injection slot을 통해 outer 4개에게 순차적으로 패킷을
주입한다. 완전한 병렬 주입을 위해서는 injection slot 수를 늘리는 추가 작업이 필요하다.

10. 의의 및 향후 과제

본 구현의 의의는 BookSim 위에서 Cross Propagation All-Reduce의 구조적 특성과 단계별 지연 특성을
직접 실험 가능한 형태로 구체화했다는 데 있다. 특히 v2에서 Phase 4를 Reduce-Scatter와 대칭적인
Px/Py broadcast 구조로 개선함으로써, 수평·수직 링크를 동시에 활용하는 균형 잡힌 통신 패턴을
구현하였다.

향후 과제로는 다음 세 가지가 중요하다. 첫째, kxk mesh에 대한 일반화가 필요하다. 둘째, injection
slot 수를 늘려 Phase 4의 broadcast를 완전 병렬화하는 구조를 검토할 필요가 있다. 셋째, processing
element의 computation delay를 포함하여 보다 현실적인 end-to-end all-reduce 모델로 확장할
필요가 있다.

이와 같은 후속 작업이 이루어진다면, 현재의 5x5 특화 구현은 향후 보다 일반적인 Cross Propagation
collective 연구를 위한 기반 모델로 활용될 수 있을 것이다.
